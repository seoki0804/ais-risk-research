PRD(Product Requirements Document)

# 문서 목적
사용자 문제, 사용 시나리오, 기능 요구사항, 비기능 요구사항, 화면 요구사항, KPI, Acceptance Criteria를 정의하여 MVP 구현 범위를 명확히 한다.

# 대상 독자
PM, UX 기획자, 데이터사이언티스트, 백엔드/프론트엔드 개발자, 연구 책임자

# 작성 버전
v1.0

# 핵심 요약
- [확정] 핵심 사용자 가치는 `자선 주변 어느 방향과 구역이 더 위험한지`를 직관적으로 파악하는 것이다.
- [확정] Must 기능은 `자선/시점 선택`, `heatmap`, `safety contour`, `speed scenario 비교`, `설명 텍스트`다.
- [합리적 가정] MVP UI는 `Notebook 기반 분석 화면`과 `경량 Web Dashboard`의 2단계 구조로 진행한다.
- [확정] 자동 제어 기능은 포함하지 않으며, 결과는 `의사결정 지원`과 `연구 분석` 목적에 한정한다.

## 1. 배경 및 문제 정의

사용자는 AIS 기반 선박 상호작용을 볼 수 있어도, 자선 주변의 공간 위험 분포를 한 번에 보기는 어렵다. CPA/TCPA 숫자만으로는 "어느 방향이 지금 더 부담스러운지"가 직관적으로 드러나지 않으며, 속력 변경 시 위험이 어떻게 이동하는지도 비교하기 어렵다.

이 PRD는 이러한 문제를 해결하는 MVP 제품 기능을 정의한다. 제품은 실시간 상용 항해 장비가 아니라, `분석형 의사결정 지원 도구`이자 `연구 데모`를 목표로 한다.

## 4. 상세 설계/요구사항/방법론

### 4.1 사용자 유형

| 사용자 유형 | 주요 목적 | 중요 기능 | 우선순위 |
|---|---|---|---|
| 항해 의사결정 지원 사용자 | 주변 위험 구역 확인 | heatmap, contour, 요인 설명 | Must |
| 해양 AI 연구자 | 규칙 기반/ML 비교 실험 | scenario 비교, export, 재현성 | Must |
| 제품 기획자/개발자 | 시스템 시연과 구조 이해 | 화면 흐름, API, 사례 replay | Should |
| 심사자/면접관 | 프로젝트 차별점 확인 | 1페이지 요약, 대표 사례 | Should |

### 4.2 사용자 문제

| 문제 ID | 사용자 문제 | 현재 한계 |
|---|---|---|
| U1 | 어느 방향이 위험한지 한눈에 보기 어렵다 | CPA/TCPA 표는 공간 직관이 약함 |
| U2 | 감속/증속 시 위험 분포 변화가 감으로만 판단된다 | 시나리오 비교 UI 부재 |
| U3 | AI 결과를 왜 믿어야 하는지 설명이 부족하다 | 블랙박스 모델은 설득력 약함 |
| U4 | 연구 데모와 제품 데모를 동시에 만들기 어렵다 | 정량성과 시각성의 균형 부족 |

### 4.3 핵심 사용 시나리오

| 시나리오 | 설명 | 입력 | 출력 |
|---|---|---|---|
| S1 | 자선과 시점을 선택하고 현재 위험도 조회 | 자선 MMSI/track, timestamp | 현재 speed scenario heatmap |
| S2 | 감속/증속 시나리오 비교 | scenario 선택 | 비교 heatmap/contour |
| S3 | 특정 고위험 셀의 원인 확인 | 셀 클릭 또는 hover | 기여 선박, 주요 factor |
| S4 | 사례를 이미지/JSON으로 export | 현재 화면 상태 | PNG/JSON/CSV |
| S5 | 연구용 배치 실행 | 날짜 범위, 해역 범위 | 요약 metric, 사례 목록 |

### 4.4 기능 요구사항

#### Must / Should / Could 우선순위

| 우선순위 | 요구사항 ID | 요구사항 |
|---|---|---|
| Must | FR-01 | 사용자가 자선 track과 시점을 선택할 수 있어야 한다 |
| Must | FR-02 | 시스템이 선택 시점 기준 주변 타선을 추출해야 한다 |
| Must | FR-03 | 시스템이 셀 단위 risk score를 계산해야 한다 |
| Must | FR-04 | 시스템이 heatmap을 지도 위에 표시해야 한다 |
| Must | FR-05 | 시스템이 safety contour를 생성/표시해야 한다 |
| Must | FR-06 | 시스템이 3개 속력 시나리오를 비교할 수 있어야 한다 |
| Must | FR-07 | 시스템이 주요 위험 요인을 설명 텍스트로 제시해야 한다 |
| Should | FR-08 | 시스템이 고위험 기여 선박 목록을 표시해야 한다 |
| Should | FR-09 | 시스템이 결과를 이미지/JSON으로 export해야 한다 |
| Should | FR-10 | 시스템이 time slider 또는 replay를 지원해야 한다 |
| Could | FR-11 | 시스템이 ML 기반 보조 위험 분류 결과를 나란히 표시할 수 있다 |
| Could | FR-12 | 시스템이 해역별 batch report를 생성할 수 있다 |

### 4.5 비기능 요구사항

| 항목 | 요구사항 | 상태 |
|---|---|---|
| Explainability | 모든 risk map은 rule contribution 또는 주요 factor를 추적 가능해야 함 | [확정] |
| Reproducibility | 동일 입력이면 동일 결과가 재현되어야 함 | [확정] |
| Performance | 주변 타선 100척 이하 기준 단일 snapshot 처리 5초 이내 목표 | [합리적 가정] |
| Usability | 지도, legend, 시나리오 차이가 직관적이어야 함 | [확정] |
| Auditability | 파라미터와 임계값을 기록/export 가능해야 함 | [확정] |
| Portability | 로컬 환경에서 실행 가능해야 함 | [확정] |

### 4.6 화면/기능별 요구사항

| 화면 | 기능 | 요구사항 |
|---|---|---|
| 메인 지도 화면 | heatmap + contour + 선박 위치 | 현재 시점의 자선 중심 위험 분포를 표시 |
| 시나리오 패널 | speed scenario 선택 | 현재/감속/증속 전환 또는 나란히 비교 |
| 설명 패널 | top drivers 표시 | 거리, TCPA, DCPA, encounter type, density 등 표시 |
| 타임라인 | 시점 이동 | 과거/미래 snapshot replay 지원 |
| Export 패널 | 결과 저장 | PNG, JSON, CSV 또는 HTML 저장 |

### 4.7 입력/출력 정의

| 구분 | 항목 | 설명 |
|---|---|---|
| 입력 | AIS raw fields | MMSI, timestamp, lat/lon, SOG, COG, heading, vessel type 등 |
| 입력 | own ship selection | 자선으로 사용할 MMSI/trajectory segment |
| 입력 | scenario params | 속력 multiplier, horizon, threshold |
| 출력 | risk map | cell별 risk score |
| 출력 | classes | 위험/주의/안전 구분 |
| 출력 | contour | iso-risk boundary geometry |
| 출력 | explanation | 고위험 방향, 기여 선박, 주요 factor |
| 출력 | export | PNG/JSON/CSV report |

### 4.8 Acceptance Criteria

| AC ID | 조건 |
|---|---|
| AC-01 | 사용자가 자선과 시점을 선택하면 주변 타선과 heatmap이 렌더링된다 |
| AC-02 | 3개 속력 시나리오 간 contour 변화가 동일 UI에서 비교된다 |
| AC-03 | 임의의 고위험 셀 선택 시 주요 기여 factor 최소 3개가 표시된다 |
| AC-04 | 결과를 PNG와 JSON 형식으로 내보낼 수 있다 |
| AC-05 | 동일 입력으로 재실행 시 동일한 risk score 결과가 생성된다 |
| AC-06 | 시스템이 결과를 `의사결정 지원`으로 설명하고 `자동 제어`로 오해되지 않게 표시한다 |

## 7. 핵심 결정사항

- [확정] MVP의 핵심 기능은 `조회`, `비교`, `설명`, `내보내기`다.
- [확정] 고급 기능보다 `해석 가능한 risk map`의 품질이 우선이다.
- [확정] 초기 제품은 실시간 운영 도구가 아니라 `이력 분석 및 데모 도구`다.
- [합리적 가정] MVP 웹 구현은 Streamlit 계열의 경량 대시보드를 우선 검토한다.
