# 문서명
연구일지 운영 가이드(Research Log Guide)

# 문서 목적
데이터 수집, 실험, 실패 사례, 모델 비교, 의사결정 변경 이력을 일관된 형식으로 기록하기 위한 연구일지 운영 규칙을 정의한다.

# 대상 독자
연구자, 데이터사이언티스트, ML 엔지니어, 지도교수

# 작성 버전
v1.0

# 핵심 요약
- [확정] 연구일지는 `좋은 결과`보다 `무엇을 왜 바꿨고 무엇이 실패했는지`를 남기는 문서다.
- [확정] 모든 실험은 `dataset_id`, `config`, `model`, `output path`, `핵심 해석`, `다음 액션`이 있어야 한다.
- [확정] case-level note와 dataset-level summary를 분리해 적는다.
- [합리적 가정] 하루에 최소 1건, 의미 있는 실험이 없으면 주간 요약 1건은 남긴다.

## 1. 배경 및 문제 정의

연구가 길어질수록 가장 먼저 사라지는 것은 "왜 이 결정을 했는지"에 대한 맥락이다. 특히 AIS 프로젝트는 데이터셋, threshold, split, own ship bundle, 모델 설정이 자주 바뀌기 때문에 메모가 없으면 같은 실수를 반복하기 쉽다.

따라서 연구일지는 단순 회고가 아니라 다음 목적을 가진 운영 문서다.

1. 재현성 확보
2. 실패의 축적
3. 논문/발표/면접용 narrative 축적

## 4. 상세 설계/요구사항/방법론

### 4.1 로그 종류

| 종류 | 목적 | 예시 파일명 |
|---|---|---|
| Daily run log | 하루 실험 기록 | `2026-03-08_baseline_bundle_a.md` |
| Dataset freeze log | 데이터셋 확정 기록 | `2026-03-09_dataset_freeze_harborA_v1.md` |
| Weekly summary | 주간 결정과 다음 주 계획 | `2026-03-13_weekly_summary.md` |
| Failure note | 실패 사례 집중 기록 | `2026-03-10_parallel_traffic_false_alarm.md` |

### 4.2 필수 항목

| 항목 | 설명 | 필수 여부 |
|---|---|---|
| 목적 | 오늘 무엇을 검증하려 했는지 | 필수 |
| dataset_id | 어떤 데이터셋인지 | 필수 |
| 입력/설정 | config, split, own ship bundle | 필수 |
| 실행 모델 | baseline/logreg/hgbt/mlp 등 | 필수 |
| 산출물 경로 | output file 또는 directory | 필수 |
| 핵심 결과 | 숫자 2~5개와 대표 해석 | 필수 |
| 실패/이상 현상 | 기대와 다른 점 | 필수 |
| 결정사항 | 무엇을 유지/폐기/보류할지 | 필수 |
| 다음 액션 | 바로 이어서 할 일 | 필수 |

### 4.3 권장 템플릿 사용 규칙

1. 새 실험을 시작할 때 먼저 로그 파일을 만든다.
2. 실험이 끝나면 결과 숫자만 적지 말고 해석을 적는다.
3. 해석이 불확실하면 `[추가 검증 필요]`로 남긴다.
4. 실패는 숨기지 말고 제목에 드러낸다.

### 4.4 좋은 로그와 나쁜 로그

| 구분 | 예시 | 평가 |
|---|---|---|
| 좋은 로그 | `drop_time_decay가 warning area를 과도하게 확장시켰고, parallel traffic에서 false alarm이 증가했다.` | 원인과 해석이 있음 |
| 나쁜 로그 | `성능이 안 좋았다.` | 재현성과 해석이 없음 |

### 4.5 권장 흐름

```text
데이터 적재 -> profile 확인 -> own ship bundle 선정 -> baseline 실행 -> comparator 실행 -> case review -> 연구일지 업데이트
```
