# 문서명
개발 로드맵 및 마일스톤 문서(Roadmap & Milestones)

# 문서 목적
개발 단계, Phase별 목표와 산출물, 완료 기준, 우선순위, 4주/8주/12주 계획을 정의해 프로젝트 실행 순서를 명확히 한다.

# 대상 독자
PM, 연구책임자, 개발자, 지도교수

# 작성 버전
v1.0

# 핵심 요약
- [확정] 로드맵은 `Phase 0 문제정의/EDA -> Phase 1 규칙 기반 baseline -> Phase 2 ML 확장 -> Phase 3 시각화 MVP -> Phase 4 평가/문서화` 순서다.
- [확정] 실제 우선순위는 `데이터와 baseline 잠금`이 가장 높고, 웹 UI와 ML은 그 다음이다.
- [합리적 가정] 1인 12주 프로젝트를 기준으로 4주/8주/12주 계획을 분리한다.
- [확정] 일정 압박 시 `ML`보다 `baseline quality + 대표 사례 시각화`를 먼저 완성한다.

## 1. 배경 및 문제 정의

이 프로젝트는 데이터 정제, 위험도 설계, 시각화, 연구 문서화가 모두 필요한 복합 과제다. 순서를 잘못 잡으면 ML 실험이나 UI 구현에 시간을 쓰고 정작 baseline 검증이 약해질 수 있다. 따라서 로드맵은 `핵심 가정 잠금 -> baseline 구현 -> 검증 -> 확장` 순서를 강제해야 한다.

## 4. 상세 설계/요구사항/방법론

### 4.1 전체 개발 단계

| Phase | 기간 | 핵심 목표 | 우선순위 |
|---|---|---|---|
| Phase 0 | 1주 | 문제 정의, 데이터 탐색, 해역/소스 확정 | P0 |
| Phase 1 | 2~4주 | 규칙 기반 baseline 구현 | P0 |
| Phase 2 | 5~7주 | ML 모델 확장 및 비교 | P2 |
| Phase 3 | 5~8주 병행 가능 | 시각화 MVP 완성 | P1 |
| Phase 4 | 9~12주 | 평가, 문서화, 논문화, 발표 자료 | P0 |

### 4.2 Phase 0: 문제 정의 / 데이터 탐색

| 항목 | 내용 |
|---|---|
| 목표 | 대상 해역, 데이터 소스, 컬럼 구조, 결측률, sample case를 확인 |
| 산출물 | EDA 노트북, 데이터 프로파일, 대상 해역 결정 메모 |
| 완료 기준 | 분석 가능한 연속 track segment 확보 여부 판단 완료 |

### 4.3 Phase 1: 규칙 기반 baseline

| 항목 | 내용 |
|---|---|
| 목표 | pairwise severity, spatial projection, contour 생성까지 구현 |
| 산출물 | risk engine 모듈, 사례별 heatmap 이미지, threshold config |
| 완료 기준 | 대표 사례 3개에서 heatmap과 contour가 해석 가능하게 출력 |

### 4.4 Phase 2: ML 모델 확장

| 항목 | 내용 |
|---|---|
| 목표 | pairwise pseudo-label 기반 logistic regression / boosting 비교 |
| 산출물 | pairwise ML baseline, 성능 표, 비교 분석 |
| 완료 기준 | rule vs ML 장단점이 문서화됨 |

주의:

- 일정 압박 시 Phase 2는 축소 가능
- baseline 품질이 낮으면 ML 착수 금지

### 4.5 Phase 3: 시각화 MVP

| 항목 | 내용 |
|---|---|
| 목표 | notebook 및 경량 web dashboard에서 scenario 비교 가능 |
| 산출물 | 데모 화면, export 기능, 설명 패널 |
| 완료 기준 | 비전문가도 사례 한 개를 보고 위험 방향을 설명 가능 |

### 4.6 Phase 4: 평가 / 문서화 / 논문화

| 항목 | 내용 |
|---|---|
| 목표 | 실험 정리, failure case 분석, 발표/포트폴리오 자료 완성 |
| 산출물 | 실험 표/그림, one-page summary, 발표 슬라이드 초안 |
| 완료 기준 | 교수/면접관 대상 설명용 서사가 완성됨 |

### 4.7 우선순위 규칙

| 우선순위 | 포함 항목 | 일정 압박 시 처리 |
|---|---|---|
| P0 | 데이터 확보, baseline, 대표 사례, 문서화 | 절대 유지 |
| P1 | 웹 UI, export, scenario panel 정교화 | 필요 시 축소 |
| P2 | ML comparator, 다중 해역 일반화 | 마지막에 착수 |

### 4.8 4주 / 8주 / 12주 플랜

#### 4주 플랜

| 주차 | 목표 |
|---|---|
| 1주 | 데이터 소스 확정, 해역 선택, EDA |
| 2주 | trajectory reconstruction, pairwise feature 생성 |
| 3주 | rule-based severity와 contour 초안 구현 |
| 4주 | 대표 사례 1~2개에서 heatmap 결과 확보 |

#### 8주 플랜

| 주차 | 목표 |
|---|---|
| 5주 | notebook 기반 scenario 비교 화면 |
| 6주 | spatial metric, ablation 초안 |
| 7주 | web dashboard MVP |
| 8주 | ML comparator 또는 추가 사례 확보 |

#### 12주 플랜

| 주차 | 목표 |
|---|---|
| 9주 | full experiment run |
| 10주 | failure case 분석, threshold calibration |
| 11주 | 논문/발표/포트폴리오 정리 |
| 12주 | 최종 데모, one-page summary, 품질 검토 |

## 7. 핵심 결정사항

- [확정] `Phase 1 baseline`이 전체 프로젝트의 게이트다.
- [확정] `ML`은 baseline과 대표 사례가 나온 뒤에만 진행한다.
- [확정] 12주 일정에서 발표/문서화는 후반 2주를 반드시 확보한다.
- [합리적 가정] 일정 지연 시 web MVP보다 notebook 결과를 우선 완성한다.
