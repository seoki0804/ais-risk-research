# 핵심 요약
- [확정] 본 프로젝트는 공개 AIS 데이터를 이용해 자선 중심의 공간 위험도(`heatmap`, `safety contour`)를 생성하는 의사결정 지원 시스템을 만드는 것이 목적이다.
- [확정] 핵심 차별점은 `pairwise risk 판단`을 `own-ship-centric spatial risk representation`으로 확장하는 데 있다.
- [합리적 가정] MVP는 `단일 혼잡 해역`, `규칙 기반 risk baseline`, `3개 속력 시나리오`, `노트북 + 경량 웹 시각화`를 기준으로 한다.
- [확정] 이 프로젝트는 `완전자율운항 제어`를 다루지 않으며, AIS만으로 가능한 수준의 위험 인지와 설명 가능한 시각화에 집중한다.

## 1. 배경 및 문제 정의

### 1.1 프로젝트 배경

AIS(Automatic Identification System)는 공개 접근이 가능한 경우가 많고, 선박의 위치, 속력, 침로 등의 정보를 포함하므로 해양 데이터 기반 프로젝트의 출발점으로 적합하다. 그러나 기존 AIS 분석은 대체로 선박 간 CPA/TCPA 계산이나 단일 pairwise encounter 분류에 머무르는 경향이 있다.

실제 항해 의사결정에서는 단순히 "어느 선박이 위험한가"를 넘어서, "자선 주변 어느 방향과 어느 공간이 상대적으로 더 위험한가"를 빠르게 인지하는 것이 중요하다. 본 프로젝트는 이 문제를 `공간 위험도 지도(spatial risk map)`로 전환한다.

### 1.2 문제 정의

| 문제 | 설명 | 상태 |
|---|---|---|
| 공간 표현의 부족 | 기존 pairwise 분석만으로는 자선 주변 전체 위험 공간을 직관적으로 보기 어렵다 | [확정] |
| 시나리오 비교의 부족 | 현재 속력 대비 감속/증속 시 위험 분포가 어떻게 달라지는지 비교하기 어렵다 | [확정] |
| 설명 가능성 요구 | 해양 도메인에서는 블랙박스 AI보다 설명 가능한 규칙이 우선 요구된다 | [확정] |
| 데이터 현실성 제약 | 공개 AIS만으로는 실제 제어 문제까지 다루기 어렵다 | [확정] |

### 1.3 왜 중요한가

- 항해 관점: 자선 주변 위험 구역을 방향성과 공간으로 이해할 수 있다.
- 제품 관점: 시각화 결과물이 분명하여 MVP 데모와 포트폴리오 가치가 높다.
- 연구 관점: `pairwise -> spatial risk mapping` 전환 자체가 연구 질문이 된다.
- 실무 관점: 규칙 기반으로도 설명 가능한 baseline을 제시할 수 있다.

### 3.3 AIS만으로 가능한 것과 불가능한 것

| 구분 | 가능 | 불가능/어려움 |
|---|---|---|
| 위험도 계산 | 거리, 속력, 방위, CPA/TCPA 기반 위험도 | 조타기/엔진 응답 기반 회피 성능 추정 |
| 시나리오 분석 | 속력 변경에 따른 상대운동 변화 분석 | 선회 의도나 사람의 의사결정 예측 |
| 시각화 | heatmap, contour, 설명 텍스트 | 실제 항해 승인 가능한 조타 명령 생성 |
| 검증 | 휴리스틱 이벤트 기반 정량 평가 | 진실 충돌확률/운항자 의도 검증 |

## 4. 상세 설계/요구사항/방법론

### 4.1 핵심 산출물

| 산출물 | 설명 | 형식 |
|---|---|---|
| 위험도 계산 엔진 | 자선 중심 셀별 risk score 계산 | Python module |
| heatmap 시각화 | 지도 위 위험도 분포 | Notebook/Web |
| safety contour | iso-risk boundary 또는 안전 경계 | 지도 overlay |
| 시나리오 비교 기능 | 현재/감속/증속 비교 | UI + export |
| 설명 텍스트 | 고위험 방향과 주요 기여 선박 설명 | text/json |
| 연구 문서 세트 | 연구계획, 실험계획, 결과 요약 구조 | Markdown |

### 4.2 범위(In Scope) / 비범위(Out of Scope)

| 구분 | 포함 항목 |
|---|---|
| In Scope | AIS 수집/정제, trajectory reconstruction, 상대운동 계산, encounter type 추정, cell risk score 계산, heatmap, contour, speed scenario 비교, 설명 가능한 baseline |
| Out of Scope | 자동 조타, collision avoidance control, 다중 센서 융합, 상용 인증, 실선 제어 검증 |

### 4.3 기대 효과

| 관점 | 기대 효과 | 상태 |
|---|---|---|
| 연구 | spatial risk mapping이라는 명확한 연구 주제 확보 | [확정] |
| 제품 | 시각적으로 설명 가능한 MVP 데모 확보 | [확정] |
| 포트폴리오 | 해양 도메인 + 데이터/ML + 시각화 역량을 동시에 보여줌 | [확정] |
| 실무 | 규칙 기반 baseline만으로도 설명 가능한 의사결정 지원 가치 확보 | [확정] |

### 4.4 일정 개요

| 단계 | 기간 | 목표 |
|---|---|---|
| Phase 0 | 1주 | 데이터 소스 확정, 해역 선정, EDA |
| Phase 1 | 2~4주 | 규칙 기반 baseline, risk score 정의 |
| Phase 2 | 5~7주 | heatmap/contour 시각화 MVP |
| Phase 3 | 8~10주 | ML 비교안, 실험, 케이스 분석 |
| Phase 4 | 11~12주 | 문서화, 발표 자료, 포트폴리오화 |

### 4.5 핵심 판단 기준

1. baseline만으로도 유의미한 heatmap과 contour가 나오는가?
2. 속력 변화에 따라 위험 분포 차이가 설명 가능하게 나타나는가?
3. 결과를 AIS-only의 한계를 넘지 않게 설명할 수 있는가?
4. 실험과 데모를 한 사람이 12주 내 구현 가능한가?

## 6. 리스크와 한계

| 리스크 | 설명 | 대응 방향 |
|---|---|---|
| 데이터 품질 편차 | AIS 누락, timestamp 불규칙, heading 결측 | 전처리 규칙과 품질 스코어 도입 |
| 해역 편향 | 특정 항만 패턴에 과적합 가능 | 단일 해역 MVP 후 일반화 검토 |
| 라벨 부족 | 충돌/근접위험의 진짜 정답 부재 | 휴리스틱 pseudo-label과 사례 분석 병행 |
| 도메인 검증 부족 | 임계값의 해양적 타당성 약화 | 전문가 리뷰 또는 공개 문헌 비교 필요 |
| 과도한 ML 기대 | baseline보다 ML 가치가 낮을 수 있음 | ML은 확장/비교안으로 제한 |

## 7. 핵심 결정사항

- [확정] 프로젝트의 중심 산출물은 `own-ship-centric heatmap`과 `safety contour`다.
- [확정] baseline은 `규칙 기반 또는 해양 도메인 기반`으로 설계한다.
- [확정] speed scenario는 `속력 변화만 반영`하고, heading 변화/복합 조종은 MVP 범위에서 제외한다.
- [합리적 가정] 초기 위험도는 `15분 horizon`, `250m cell`, `3개 scenario` 기준으로 설계한다.
